Cross-Cutting — Decision Capture

What Is an Architecture Decision Record?

An Architecture Decision Record (ADR) is a short, version-controlled document that captures one load-bearing decision — the context that forced it, the options considered, the option chosen, and the consequences accepted. ADRs are the only artifact in the development stack that distinguishes essential constraints from accidental ones. Without them, code, tests, and design documents all flatten into “this is how it is” and a future maintainer (human or AI) cannot tell which parts of the system are load-bearing and which are incidental.

The Threshold

An ADR is created when all of the following hold:

  • Load-bearing. Changing the decision later could break the system or invalidate downstream design.
  • Multiple real options existed. Two or more alternatives were on the table and one was picked.
  • Rests on assumptions or drivers that may change. The decision is contingent on something specific to the present context.

An ADR is not created for routine choices, taste calls, or the mechanical application of an established pattern. The threshold is deliberately high: ADRs that are written for everything stop being written at all (the capture-cost master constraint — see References).

Self-test: “Could a future maintainer who changes this break the system?” — if yes, ADR; if no, no ADR.

The ADR Is the Artifact of the Decision Conversation

An ADR is not a justification written after a design is drafted. It is the record of the conversation in which the decision was made. The order is:

  1. A conversation (human-AI, or human-human) surfaces options and weighs them against drivers.
  2. A decision is reached.
  3. The ADR is drafted capturing that decision.
  4. The design derives from the ADR.

If the design is drafted first and the ADR is written afterwards to justify it, the option space has already collapsed and the ADR is reconstruction rather than recording. The same failure mode — confident invention rather than faithful capture — that makes legacy rationale recovery prone to fabrication also corrupts post-hoc ADRs in greenfield work.

What an ADR Is Not

  • Not a design document. An ADR captures a decision; the design captures the structure that implements it. The two are linked but distinct.
  • Not a meeting record. Discussion is summarised — the decision and the reasons are what survive, not the conversation transcript.
  • Not a justification for a chosen technology. If only one option was ever considered, there is no decision and no ADR — just a note in the design.
  • Not a retrofitted record. Reconstructing rationale for legacy code is forbidden. See §3 Forward-Only Policy.

ADRs in the V-Model

ADRs are cross-cutting. They are not owned by a single V-model layer. A load-bearing decision can occur at any layer at which design happens — system architecture, software architecture, detailed design, or component-level when a load-bearing pattern is selected. The threshold filters which decisions become ADRs; the layer just labels them.

Where ADRs Originate

Two sources, both legitimate:

  • Research / plan phase. The interactive conversation in which a human and an AI agent shape the approach to a piece of work. Decisions reached during this phase that meet the threshold become ADRs as part of the plan output.
  • Build phase, via HALT. A developer agent executing a task contract may encounter a load-bearing decision it cannot make alone. It writes a HALT artifact with category design-issue; the resolving human-AI conversation produces the ADR; the agent resumes with the ADR + updated design as input. See HALT Pattern in the Skills Architecture.

Trigger: Either Side

Both human and AI may propose an ADR. Each shares the discipline:

  • The AI is instructed (in skill prompts) to recognise when a load-bearing decision is being made and suggest capturing an ADR.
  • The human exercises the same recognition independently and may call for an ADR even when the AI did not suggest one.

The decision to accept an ADR is always the human's. AI may draft the body; only a named human appears in deciders.

Producing a Quality ADR

Quality, for an ADR, is not literary. The ADR is good when a different engineer can read only the ADR and derive the same design from it. If they cannot, the ADR has under-specified the decision and the design contains commitments that were not actually captured.

Template

Markdown with YAML frontmatter. One file per decision under docs/artifacts/adr/ADR-nnn.md. The full schema is in schemas/artifacts/adr.schema.yaml; the template content is reproduced here for reference.

Frontmatter

---
artifact_id: ADR-001
artifact_type: adr
title: "Token bucket for fuel-rate limiting"
status: accepted              # proposed | accepted | superseded
deciders: ["Stefanus Bergenhem"]
date: 2026-04-17
provenance: recorded          # only legal value — see Forward-Only below
layer: detailed-design        # optional; system-arch | sw-arch | detailed-design | component | cross-cutting
related:
  requirements: [SWREQ-042, SWREQ-044]
  produces: [DD-017]
  # supersedes: ADR-012       # when this replaces an earlier decision
---

Body (required headings, in order)

# ADR-001: Token bucket for fuel-rate limiting

## Context and Problem Statement
2-5 sentences on what changed, why a decision is needed now,
and the forces at play. Neutral language, no solutions yet.

## Key Assumptions
- Bullet list. Grounds on which this decision rests.
- If any of these change, the ADR may need to be revisited or superseded.

## Decision Drivers
- Bullet list. Criteria, quality attributes, or constraints
  that weigh on the choice.

## Considered Options
- Option A — short description
- Option B — short description
- (At least two. A single-option ADR is a description, not a decision.)

## Decision Outcome
Chosen option: B, because [justification tied explicitly to named drivers].
One short paragraph.

## Positive Consequences
- Bullet list of expected benefits.

## Negative Consequences (Trade-offs)
- Bullet list of accepted costs, risks, or constraints.
- An ADR with no negatives listed is almost certainly missing them.

The Conversation Pattern

A typical run from problem to ADR:

  1. Surface options. Either side names two or more candidate approaches. The AI is good at brainstorming; the human is good at filtering for context-fit.
  2. Name the drivers. Quality attributes, constraints, deadlines, dependencies. Make them explicit before evaluating options — otherwise post-hoc weighting creeps in.
  3. State assumptions. What does each option assume about the world? These are the future revisit triggers; surfacing them early makes them inspectable.
  4. Pick. The decision emerges from how options score against drivers. The justification line in the ADR cites the drivers by name.
  5. Capture, then design. Write the ADR. Then derive the design from it — not the other way around.

Forward-Only: Never Reconstruct Rationale

An ADR records a decision at decision time. Reconstructing rationale for legacy code is forbidden by this framework. The provenance field has exactly one legal value, recorded, precisely so that there is nowhere to put “reconstructed.” The schema itself enforces the policy.

Why: an LLM (or a human guessing from code) asked to recover the original rationale produces confident invention — plausible-sounding justifications, fabricated constraint sources, false-precision trade-offs. The failure mode is not silence (which would be detectable) but confident invention (which looks like rationale and isn't). Once invented rationale enters the trace graph, every downstream artifact that cites it inherits the contamination.

Honest alternative for legacy: mark the decisions “rationale unknown” in whatever index the project uses. Begin the ADR stream from baseline-day forward. Pretending to recover rationale is worse than admitting it is gone.

Fitness Test

One question, applied after the ADR is drafted:

“Could a different engineer, reading only this ADR, derive the same design from it?”

  • If yes — the ADR is sufficient.
  • If no — the ADR under-specifies the decision; the design contains commitments that were not actually captured. Revise the ADR before accepting it.

Common Anti-Patterns

  • Single-option ADR. Lists one option and a justification. This is a description, not a decision — either rewrite with the alternatives that were actually considered, or skip the ADR entirely.
  • Generic justification. “Chosen because it is more flexible / more modern / industry standard.” The justification must reference the named drivers from this ADR — not generic praise of the option.
  • Missing negatives. Positive consequences listed; negatives empty or hand-waved. Every load-bearing decision has trade-offs; if you cannot name them, you have not understood the decision.
  • Buried assumptions. Assumptions implicit in Context prose rather than listed in Key Assumptions. Future maintainers will miss them.
  • Post-hoc rationalisation. Design drafted first, ADR written to match. Detectable by the absence of any options that would have produced a different design — the option space has already been pruned to one.
  • ADRs for routine choices. Capturing every variable name and method signature as an ADR. Ratio of signal to noise collapses; nothing gets read.

V-Model Considerations

Traceability

ADRs are first-class trace nodes. Other artifacts cite an ADR by its artifact_id. Trace links are managed by the Traceability pillar (Pillar 2), not embedded in the ADR body. The related frontmatter is a lightweight in-document mirror of the formal trace links.

FromToMeaning
RequirementADRThe decision constrains or interprets the requirement
Design artifactADRThe design implements the decision
ADRADRSupersedes / refines

The interesting open question — whether ADRs sit in the standard trace chain (requirement → ADR → design → code → test) or as a parallel rationale graph anchored to design elements — is being worked separately. See backlog item under Component 2.

Immutability and Supersession

An accepted ADR is append-only. Decisions change; the historical record does not. To change a decision:

  1. Write a new ADR. Set its status to accepted and its related.supersedes to the old ADR's id.
  2. Update the old ADR: change status to superseded; set related.superseded_by to the new ADR.

Never edit the body of an accepted ADR. Preserving the decision graph is what makes informed change possible later — including the ability to revisit a decision when one of its key assumptions turns out to have changed.

Build-Phase Integration via HALT

The HALT pattern in the Skills Architecture defines a design-issue category for when an agent encounters a load-bearing decision it cannot make alone. The flow:

  1. Agent hits the design issue — writes halt.yaml with category: design-issue and the candidate options listed.
  2. Human is notified, opens a conversation with the agent (or a fresh one) to resolve.
  3. Resolution produces an ADR. The HALT and the ADR are separate artifacts — the HALT records the pause; the ADR records the decision.
  4. Agent resumes with the ADR and the updated design as input.

This is the only sanctioned way for build-phase decisions to enter the trace graph. An agent that makes a load-bearing decision unilaterally and continues is in violation of the framework discipline; build agents are instructed to HALT.
