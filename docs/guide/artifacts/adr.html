1. V-Model Context

An Architecture Decision Record (ADR) is a short, version-controlled document that captures one load-bearing decision: the situation that forced it, the options considered, the option chosen, the reasoning, and the consequences accepted. One decision per file.

Cross-cutting by design

ADRs are not bound to a single layer of the scope tree. A load-bearing decision can surface at any depth — the root product, an intermediate subsystem, or a leaf module. The scope_tags field names the scope(s) the decision primarily applies to; ADRs themselves live flat in /specs/adrs/ so that a decision that spans multiple children is not buried inside one of them.

Why the framework treats ADRs as first-class

Code, tests, and state documentation together describe what the system is. They do not distinguish essential constraints — the ones a change must respect — from accidental ones, which just happened to be how someone typed it. Every line of code looks equally load-bearing without a record of which constraints carry weight and why.

Rationale is the artifact that draws that line — the essential-vs-accidental constraint distinguishability that the rest of the artifact set depends on. When it is captured at decision time, a future reader (human or agent) can tell a hard requirement from an incidental choice, safely refactor the accidental parts, and negotiate the essential parts with eyes open. When rationale is missing, every refactor looks risky and every review is reduced to checking that code matches tests. Tests without rationale cannot explain themselves: a passing assertion may encode a specification, a customer accommodation from 2019, or a hardware quirk, and all three look identical in the test file.

The threshold for authoring an ADR

All three conditions must hold. If any one is missing, it is not ADR-worthy — note it inline in the Architecture or Detailed Design and move on.

  • Load-bearing. Changing the decision later could break the system or invalidate downstream design.
  • Multiple real options existed. Two or more alternatives were genuinely on the table.
  • Contingent on assumptions or drivers that may change. The decision rests on something specific to the present context; if those inputs shift, the decision may need to be revisited.

Deliberately high bar. ADRs written for every naming convention and import choice stop being written for the decisions that matter.

What an ADR is not

  • Not a design document. The ADR captures a decision; the Architecture or Detailed Design captures the structure that implements it.
  • Not a meeting record. Discussion is summarised to the decision and the reasons that survive it. Transcripts do not belong.
  • Not a retrofit invention. When rationale for a pre-existing decision is lost, the framework records that it is lost, not a plausible-sounding story.

Where ADRs originate

Default origin: during Architecture authoring, as the decomposition exposes a decision that meets the threshold. The Architecture references the ADR through governing_adrs.

Alternative origin: before Architecture, when an external or organisational decision already binds — e.g., "we run on Postgres" is policy before this scope exists. The ADR is written first, and the Architecture consumes it through the same governing_adrs link.

2. Best Practices

Canonical fields

An ADR is a single Markdown file under /specs/adrs/adr-NNN-{slug}.md with YAML front-matter and a short body. The canonical field list:

FieldPurpose
idStable identifier: ADR-{seq}-{slug}.
titleShort active-voice noun phrase naming the decision.
statusproposed → accepted → superseded.
dateDate of acceptance (or first draft if still proposed).
scope_tagsScope(s) the decision primarily applies to. Non-empty.
supersedes / superseded_byADR lineage when this replaces or is replaced by another.
affected_scopesOptional: scopes the decision affects beyond its primary scope.
contextThe situation addressed. What forces are at play, and why now.
decisionWhat was chosen. Active voice ("We will ...").
alternatives_consideredAt least two. Each with a concrete reason for rejection.
rationaleWhy the chosen option wins against the named drivers.
consequencesPositive and negative outcomes. Ends with the Reversibility sub-prompt (below).

The Reversibility sub-prompt — mandatory inside Consequences

Every ADR's Consequences section ends with a direct answer to this prompt, verbatim:

Is this decision reversible? If yes: state the rollback path. If no: state the recovery plan and name who must sign off before implementation.

Reversibility sets the risk profile. A reversible decision can be walked back with a migration or a feature flag — review it like any other technical choice. An irreversible decision persists in data shape, public contracts, or regulatory posture. It deserves a named human on the hook and a recovery plan written before implementation, because the cost of being wrong is not a sprint; it is a rewrite.

The prompt is answered, not acknowledged. "Partially reversible" is not an answer — name which parts are which, and treat each on its own terms.

Conversation pattern — capture then design

The sequence that produces a faithful ADR:

  1. Surface options. Two or more real candidates, named.
  2. Name the drivers. Quality attributes, constraints, deadlines, dependencies — made explicit before any option is evaluated, so weighting does not creep in post-hoc.
  3. State assumptions. What each option assumes about the world. These are the revisit triggers; surface them so they can be inspected.
  4. Pick. The chosen option scores best against the drivers. The rationale cites drivers by name.
  5. Capture. Write the ADR.
  6. Design. Derive the Architecture or Detailed Design from the ADR, not the other way around.

If the design is drafted first and the ADR is written afterwards, the option space has already collapsed. What remains is reconstruction, and reconstruction looks like rationale without being rationale.

Capture at decision time

Rationale is lossy the moment a decision is made. The options that were on the table, the trade-offs that were weighed, the assumptions that pointed one way rather than another — all of it becomes invisible in the resulting artifact. A week later, the author can reproduce an approximation; a month later, a plausible-sounding version; a year later, a story that reads well but does not match what happened.

Write the ADR while the conversation is still in working memory. This is cheap at decision time and expensive at any later point.

Immutability and supersession

An ADR's lifecycle is proposed → accepted → superseded. Once accepted, the body is immutable. Changing a decision does not mean editing the old ADR; it means writing a new one that supersedes it:

  1. Author a new ADR. Set its status to accepted and its supersedes to the old ADR's id.
  2. Edit the old ADR's front matter only: set status to superseded and superseded_by to the new ADR's id. Do not touch the body.

The point is the historical graph. A maintainer revisiting a decision three years later needs to see the chain — what was decided, what it was replaced with, and why — not a single document that has been silently rewritten.

Propagation — hybrid rule

An ADR's consequences create downstream obligations. The framework propagates these in two ways, and one ADR can use both:

  • New requirement at the ADR's own layer — when the consequence is testable at that layer. Example: "latency < 50 ms at the queue ingress" becomes a Quality Attribute requirement on the scope that owns the queue.
  • governing_adrs reference from child artifacts — when the consequence only bounds choices made lower down. Example: "job payloads are JSON" propagates as a constraint the child Detailed Design honours, not a testable target at the ADR's layer.

Completeness rule

Every consequence must be either (a) satisfied by a requirement at the ADR's scope, or (b) referenced as a constraint by a child artifact. Tooling enforces this. A consequence that sits in the ADR and does not surface anywhere downstream is a dropped obligation — the decision said the system would behave a certain way and nothing in the spec tree commits to it.

The Y-statement compact form

When the decision fits it, a Y-statement lets a skilled reader absorb the ADR in one sentence:

In the context of {situation}, facing {concern}, we decided for {option}, to achieve {quality}, accepting {downside}.

Use it as an abstract line at the top of the ADR, not as a replacement for the fields. The five anchors map to context, drivers, decision, rationale, and the chief negative consequence; the fields still need to stand up to the Quality Bar.

Retrofit posture — no fabrication

When an ADR captures a pre-existing decision whose rationale is lost, the human-only fields carry recovery_status: unknown:

  • context, alternatives_considered, rationale, and anticipated consequences are human-only. Code does not contain them.
  • The only legal states for these fields in retrofit are verified (a human supplied the content from memory or archive) and unknown (no honest record exists).
  • There is no reconstructed state for these fields. AI inference into human-only fields is refused at the skill level, not left to agent discretion.

An ADR with unknown human-only fields is the correct retrofit outcome when rationale is gone. It is honest, it says what is known (the decision, visible in the code), and it does not launder the current state as inevitable.

3. Anti-Patterns

  1. Single-option ADR. One option listed, followed by a justification. This is a description of what was done, not a decision. Tell: the "alternatives" section is empty, absent, or filled with straw men like "do nothing."
  2. Generic justification. The rationale reads "more flexible," "more modern," "industry standard," or "best practice." Tell: the same sentence could be pasted into twenty unrelated ADRs without editing. The rationale must cite the drivers named in this ADR, by name.
  3. Missing negatives. Consequences list only upsides. Every load-bearing decision trades something away. Tell: the Consequences section reads as marketing copy; the cost side is empty or hand-waved ("some additional complexity").
  4. Buried assumptions. Assumptions are implicit in the Context prose rather than enumerated. Future maintainers will miss them. Tell: the ADR has no place a reader can point to and say "if this changes, revisit."
  5. Post-hoc rationalisation. The design was drafted first; the ADR was written afterwards to match. The option space is pruned to one before the ADR starts. Tell: the rejected alternatives would all have produced approximately the same design, which is another way of saying they were never real.
  6. ADRs for routine choices. Variable names, method signatures, directory layouts. The signal-to-noise ratio of the ADR stream collapses and the decisions that matter stop being read. Tell: more than one or two ADRs per scope per sprint in steady state.
  7. Test-as-requirement inversion (retrofit). A characterization test's assertion is read as the original intent, and the ADR is written to explain why the intent is the intent. Tell: the "rationale" reads as a paraphrase of the test expectation; the Context does not describe the forces that would have produced the decision.
  8. LLM confident invention (retrofit). An agent asked to reconstruct rationale for a pre-existing decision emits crisp, committee-grade prose with named alternatives and neat rejection reasons — all fabricated. Particularly insidious because the output looks like a good ADR. Tell: the retrofit had no preserved conversation, no archive, no deciders alive to interview, yet the ADR reads as if minutes were taken.
  9. Laundering the current state (retrofit). The ADR reads as a post-hoc defence of the present design, explaining why the current approach is the right one — rather than recording what was actually decided and why. Tell: every alternative is rejected for a reason that happens to be a property the current design has.
  10. Missing Reversibility answer. The Consequences section is present but the Reversibility sub-prompt is left unanswered, acknowledged with a stock phrase, or hedged into uselessness ("somewhat reversible"). Tell: no rollback path for a reversible decision, or no named sign-off for an irreversible one.
  11. Orphan ADR. No scope_tags set, and no artifact references it via governing_adrs. The decision exists in the ADR directory and nowhere else. Either the decision is not actually load-bearing (delete it) or the propagation rule has been skipped (wire it up).

4. Examples

Good example — fully populated

A web-service scope facing a choice of mechanism for its asynchronous job queue. The team considered three real options, picked one, and recorded the decision before sketching the queue's design.

---
id: ADR-017-use-postgres-for-job-queue
title: "Use Postgres for the asynchronous job queue"
status: accepted
date: 2026-03-12
scope_tags: [app/jobs]
affected_scopes: [app/jobs, app/ops]
---

# ADR-017: Use Postgres for the asynchronous job queue

**Y-statement.** In the context of the app's background job processing, facing
the need for at-least-once delivery with operationally cheap retries, we
decided to use Postgres (via `SKIP LOCKED`) as the queue backing store, to
achieve transactional job enqueue with the same durability guarantees as the
rest of our data, accepting lower peak throughput than a dedicated broker.

## Context
The app already writes its primary domain state to Postgres. Several features
require side effects that must happen after a request commits (e.g., sending
receipts, provisioning downstream accounts). We need a way to schedule and
retry these jobs durably. The ops team manages one Postgres cluster today and
has no experience operating Redis or RabbitMQ in production.

## Decision
We will store queued jobs in a Postgres table and dequeue with
`FOR UPDATE SKIP LOCKED` from a small fleet of worker processes.

## Alternatives considered
- **Redis with a Lua-scripted reliable queue.** Rejected: introduces a second
  stateful system to operate; ops familiarity is zero; durability story is
  weaker without AOF tuning the team does not currently do.
- **RabbitMQ.** Rejected: delivers the feature set but adds broker operations
  (clustering, disk sizing, upgrade dance) that the ops team cannot take on
  this quarter; the throughput headroom is not needed at current load.
- **Cloud-managed queue (SQS / Pub-Sub).** Rejected: couples the app to a
  specific cloud vendor; we have a portability requirement from ADR-003.

## Rationale
Two drivers decided it: (1) **operational familiarity on the ops team** —
adding Postgres tables is free, adding a new stateful system is not, and (2)
**ACID enqueue for idempotency-critical retries** — jobs enqueued in the
same transaction as the domain write cannot diverge from it, which Redis and
RabbitMQ can only approximate with outbox patterns.

## Consequences
**Positive**
- Single source of durability for domain state and job state; no dual-write
  inconsistency; no outbox pattern needed.
- Ops surface does not grow.
- Jobs are visible via SQL for debugging and reporting.

**Negative**
- Peak throughput ceiling (~2k jobs/sec on our current hardware) is lower
  than a dedicated broker would provide. Above that, we revisit.
- Long-running worker transactions can interact badly with autovacuum; we
  accept this and constrain job handlers to short transactions.

**Reversibility.** Reversible. Rollback path: workers read via a queue-client
adapter (see DD-app-jobs-worker); swapping to Redis or a broker means
implementing a second adapter behind the same interface and running both in
parallel for one release while draining the Postgres queue. Migration cost
estimated at ~1 engineer-week; no data-loss risk because queued jobs are
derivable from domain events.

## Propagation
- New requirement at this scope: **REQ-app-jobs-018** — *"Job enqueue
  MUST occur in the same database transaction as the domain write that
  produces the job."* Testable at `app/jobs` integration level.
- Constraint on child designs: worker implementations reference this ADR
  via `governing_adrs: [ADR-017]` and use the `SKIP LOCKED` dequeue pattern.

What this example gets right: two drivers cited by name in rationale (not generic praise); three alternatives with concrete, context-specific rejection reasons; explicit negatives with a threshold at which the decision is revisited; Reversibility answered with a real rollback path and a rough migration cost; a new testable requirement materialised at the ADR's layer; a clear propagation path to child designs.

status: accepted
date: 2026-03-12
scope_tags: [app/jobs]
---

# ADR-042: Use Redis for async jobs

## Context
We need an async job queue for the app.                         ← No forces, no drivers, no "why now."

## Decision
We will use Redis because it is more modern and flexible.       ← Generic justification; drivers not named.

## Alternatives considered
- Don't use a queue.                                            ← Straw man. Single real option.

## Rationale
Redis is the industry standard for this kind of workload.       ← Could be pasted into any ADR, unchanged.

## Consequences
- Fast.                                                         ← Only upside; negatives missing.
- Developers are happy with it.                                 ← Not a consequence of the system.

                                                                ← Reversibility sub-prompt missing entirely.

Four anti-patterns in one page: single-option (the alternatives list is a straw man), generic justification, missing negatives, missing Reversibility answer. A reader has no idea which constraints are load-bearing, what breaks if the ops team can't run Redis, or whether switching back is an afternoon or a rewrite.

5. Quality Bar

Yes/No checklist, grouped by concern. An ADR passes when every applicable item is answered Yes. The meta-gate at the bottom — the Spec Ambiguity Test — overrides all of them: an ADR that ticks every box but cannot guide a junior engineer to the same design fails.

Context completeness

  • Does the Context describe the specific situation that forced the decision (not a generic statement of the problem domain)?
  • Are the forces at play — constraints, deadlines, dependencies — named?
  • Are the decision drivers explicit, so the rationale can cite them by name?

Option space integrity

  • Are at least two real alternatives listed? ("Do nothing" and straw men do not count.)
  • Does each alternative have a concrete, context-specific reason for rejection?
  • No single-option smell — would any of the rejected options have produced a materially different design?

Decision clarity

  • Is the Decision stated in active voice, with the chosen option named?
  • Does the Rationale cite the drivers from this ADR, by name — not generic praise of the option?

Consequences discipline

  • Are positive consequences listed?
  • Are negative consequences listed, non-empty, and concrete?
  • Is the Reversibility sub-prompt answered — either with a rollback path (reversible) or with a recovery plan and a named sign-off (irreversible)?

Linkage

  • Is scope_tags non-empty?
  • If this ADR supersedes or is superseded, is the chain intact in both directions?
  • If the decision affects scopes beyond scope_tags, is affected_scopes set?

Completeness rule

  • Is every consequence either satisfied by a requirement at the ADR's scope or referenced as a constraint by a child artifact? (No orphan consequences.)

Propagation rule

  • For consequences testable at this ADR's layer: has a new requirement been materialised at that layer?
  • For consequences that only bound child choices: are the relevant child artifacts linked via governing_adrs?

Retrofit honesty

  • If this is a retrofit of a pre-existing decision and the rationale is lost: is recovery_status: unknown set on Context, Alternatives, and Rationale?
  • Are all fields free of AI-fabricated content — no committee-style prose for conversations that did not happen, no tidy rejection reasons invented from the current code?

Spec Ambiguity Test — meta-gate

  • Could a junior engineer or low-mid-tier AI, reading only this ADR, derive the same design from it without guessing?

If the answer is No, the ADR under-specifies the decision. Revise before accepting. This test overrides every box above: an ADR that passes all of them but fails this one is not done.
